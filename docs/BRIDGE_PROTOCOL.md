# HubEx Bridge Protocol v1

## Overview

The HubEx Bridge Protocol enables an ESP32 (with WiFi) to act as a WiFi bridge for non-internet-capable microcontrollers (Arduino, ATmega, STM32). Communication is via UART (Serial) with a simple text-based protocol.

## Architecture

```
[Arduino/ATmega] ←UART→ [ESP32 Bridge] ←WiFi→ [HubEx Backend]
```

The ESP32 handles:
- WiFi connection + HubEx pairing
- Telemetry forwarding (Arduino → HubEx)
- Variable delivery (HubEx → Arduino)
- Heartbeat + OTA for the ESP32 itself
- Remote flash for the Arduino (via STK500 protocol)

## Protocol Specification

### Message Format

```
<COMMAND> <KEY> <VALUE>\n
```

- Commands are uppercase, 3-4 characters
- Keys are alphanumeric + dots/underscores
- Values are strings (JSON for complex types)
- Messages terminated by `\n`
- Max message length: 256 bytes

### Commands

| Command | Direction | Format | Description |
|---------|-----------|--------|-------------|
| `VAR` | Arduino → ESP | `VAR temperature 23.5` | Report a variable value |
| `SET` | ESP → Arduino | `SET led_state 1` | Set a variable on the Arduino |
| `ACK` | Both | `ACK SET led_state` | Acknowledge a command |
| `NACK` | Both | `NACK SET led_state invalid` | Reject a command with reason |
| `PING` | ESP → Arduino | `PING` | Heartbeat check |
| `PONG` | Arduino → ESP | `PONG` | Heartbeat response |
| `META` | Arduino → ESP | `META name "My Sensor"` | Report device metadata |
| `META` | Arduino → ESP | `META version 1.0.2` | Report firmware version |
| `RDY` | Arduino → ESP | `RDY` | Arduino ready after boot |
| `RST` | ESP → Arduino | `RST` | Request Arduino reset |
| `FW` | ESP → Arduino | `FW BEGIN 1234` | Start firmware upload (bytes) |
| `FW` | ESP → Arduino | `FW DATA <base64>` | Firmware data chunk |
| `FW` | ESP → Arduino | `FW END <checksum>` | End firmware upload |

### Example Session

```
# Arduino boots
RDY
META name "Workshop Sensor"
META version 1.0.0

# ESP responds
ACK META

# Arduino reports values
VAR temperature 23.5
VAR humidity 67.2
VAR light_level 850

# ESP forwards to HubEx, receives command
SET relay_1 1
# Arduino processes
ACK SET relay_1

# Periodic heartbeat
PING
PONG
```

### Error Handling

- If no PONG received within 5s → retry 3x → mark Arduino as offline
- If ACK not received within 2s → retry once
- Checksum on FW DATA: CRC16 per chunk
- Buffer overflow protection: max 256 bytes per message

## ESP32 Bridge Firmware Template

The bridge firmware is auto-generated by HubEx and handles:

1. WiFi connection (credentials from SPIFFS/NVS)
2. HubEx pairing (hello → claim → confirm → token)
3. Serial bridge (parse incoming VAR/META, forward SET)
4. Telemetry batching (collect VARs, send every N seconds)
5. Heartbeat (PING/PONG every 30s)
6. OTA updates for the ESP32 itself
7. Remote Arduino flash via STK500 (optional)

## Arduino Library

```cpp
#include <HubExBridge.h>

HubExBridge bridge(Serial);  // Use hardware serial

void setup() {
  Serial.begin(115200);
  bridge.begin();
  bridge.setMeta("name", "My Sensor");
  bridge.setMeta("version", "1.0.0");
}

void loop() {
  bridge.loop();  // Process incoming commands

  // Report values
  bridge.setVar("temperature", readTemp());
  bridge.setVar("humidity", readHumidity());

  // Check for incoming commands
  if (bridge.hasCommand("relay_1")) {
    digitalWrite(RELAY_PIN, bridge.getInt("relay_1"));
    bridge.ack("SET", "relay_1");
  }

  delay(1000);
}
```

## Configuration

The bridge is configured via HubEx Device Detail → Config panel:

```json
{
  "bridge_mode": true,
  "baud_rate": 115200,
  "target_board": "atmega328",
  "remote_flash": true,
  "telemetry_interval_seconds": 10,
  "heartbeat_interval_seconds": 30
}
```
